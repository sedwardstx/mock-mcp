# Story 2.3: Ticket Search & Filter Tool

## Status

Draft

## Story

**As a** student's AI agent,
**I want** to search and filter tickets by their attributes,
**so that** I can find the specific incident(s) relevant to a lab or investigation.

## Acceptance Criteria

1. A `search_tickets` tool supports filtering by attributes including Azure product/service, status, severity, customer persona (Windows admin vs. Azure developer), and affected resource.
2. Filters are combinable (AND semantics) and return a bounded, paginated summary result set.
3. A search matching no tickets returns an empty result set (not an error), and invalid filter values return a clear validation message.
4. The tool exposes a clear MCP schema documenting each supported filter and its allowed values.
5. Integration tests assert deterministic search results for representative filter combinations against known sample scenarios.

## Tasks / Subtasks

- [ ] Task 1: Repository search (AC: 1, 2)
  - [ ] `Repository.search_tickets(persona, status, severity, azure_product, resource_id, offset, limit) -> (tickets, total)` with AND semantics over the deterministic Dataset order
- [ ] Task 2: `search_tickets` tool (AC: 1, 3, 4)
  - [ ] Optional filter params; validate enum filters (persona/status/severity) against allowed values → structured `invalid_request` with message
  - [ ] No matches → `ok` with empty list; document allowed values in the tool description
- [ ] Task 3: Tests (AC: 5)
  - [ ] Unit: repository search (single + combined filter, no-match)
  - [ ] Integration: search by persona, by status, combined; no-match empty; invalid filter value → invalid_request

## Dev Notes

### Filter Design [Source: prd/requirements.md] (FR4) [Source: architecture/components.md#Repository]
- Filterable attributes: `azure_product` (exact), `status` (enum), `severity` (enum), `persona` (enum), `resource_id` (membership in ticket.resource_ids). AND semantics.
- Reuse `TicketListResult`/`TicketSummary` from Story 2.2; add an optional `message` field for validation feedback.
- Enum allowed values come from `data/models.py` (`Persona`, `Status`, `Severity`).

### Error Handling [Source: architecture/error-handling-strategy.md#Business-Logic-Errors]
- Invalid enum filter value → structured `invalid_request` with allowed values; never raise.
- Empty result is `ok` with `tickets: []` (NOT an error). [Source: prd/requirements.md] (FR- empty result behavior)

### File Locations [Source: architecture/source-tree.md]
- `src/contoso_support_mcp/data/repository.py` (extend)
- `src/contoso_support_mcp/tools/tickets.py` (extend — add `search_tickets`, add `message` to result model)
- Tests: `tests/unit/test_repository.py` (extend), `tests/integration/test_ticket_tools.py` (extend)

### Previous Story Insights (2.2)
- `Repository` and ticket tools already exist; extend them (do not create parallel modules).
- Structured-result contract established — follow it exactly.

### Testing
- pytest + pytest-asyncio; in-memory MCP client. Assert deterministic filtered results against the 3 shipped samples (personas: windows_admin ×2, azure_developer ×1; products: VMs ×2, Networking ×1).
- **Run:** `uv run pytest`.
