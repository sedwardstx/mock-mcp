# Story 3.4: Compute Log Query Tools (Host & Windows Guest, VM & VMSS)

## Status

Approved

## Story

**As a** student's AI agent,
**I want** to query Compute platform-host logs and Windows guest logs for VMs and VMSS,
**so that** I can investigate both the Azure host layer and the in-guest Windows behavior behind an incident.

## Acceptance Criteria

1. Tools are provided to query **Compute host/platform logs** and **Compute Windows guest logs**, each supporting both **VM** and **VMSS** resources (including addressing an individual instance within a VMSS).
2. Each tool accepts a resource identifier, an optional time range, and compute-appropriate filters, and returns matching rows scoped to that resource/instance and time range.
3. Results are deterministic for identical inputs and correlated to the scenario — compute-root-caused scenarios contain supporting evidence across host and/or guest logs, with realistic noise.
4. Guest logs are Windows-only, consistent with the MVP scope.
5. Empty results return an empty set; malformed inputs (e.g., a VMSS instance that doesn't exist) return a clear validation message.
6. Each tool exposes a clear MCP schema documenting parameters, filters, and returned columns.
7. Integration tests assert correct, deterministic results for representative VM and VMSS scenarios across host and guest logs.

## Tasks / Subtasks

- [ ] Task 1: Resource lookup by id (AC: 5)
  - [ ] `Repository.get_resource(resource_id) -> AzureResource | None` (scan scenarios' resources)
- [ ] Task 2: `query_compute_host_logs` + `query_compute_guest_logs` tools (AC: 1, 2, 4, 6)
  - [ ] Params: resource_id (req), time_range (opt), instance_id (opt, VMSS), plus one compute filter (host: event_name / level; guest: level)
  - [ ] If instance_id given and resource is known but instance not in `resource.instances` → `invalid_request` with allowed instances
- [ ] Task 3: Backfill compute telemetry samples (AC: 3)
  - [ ] TICKET-10000003 (guest, VM): Windows event 7031 evidence + benign noise
  - [ ] TICKET-10000008 (host, VMSS): host degraded event on instance data-vmss_0 + healthy row for data-vmss_1
- [ ] Task 4: Tests (AC: 3, 5, 7)
  - [ ] Integration: guest query on VM → event 7031; host query on VMSS instance → degraded row; unknown instance → invalid_request; empty for unrelated resource

## Dev Notes

### Design [Source: architecture/database-schema.md#Table-ComputeHostLogs] [Source: architecture/database-schema.md#Table-ComputeGuestLogs]
- Two tools over `compute_host_logs` / `compute_guest_logs` via the shared `_query` helper (Story 3.2). Add `instance_id` scoping (already supported by `query_rows`).
- **Instance validation (AC5):** when `instance_id` is provided, look up the resource; if known and `instance_id not in resource.instances` → `invalid_request` listing valid instances. Unknown resource → empty `ok` (consistent with other tools).
- Host columns: instance_id, host_node, event_name, health_status, maintenance_type, level, message. Guest columns: instance_id, computer, channel, provider_name, event_id, level, task, message. Guest is Windows-only.
