# Story 1.1: Project Scaffold & Health Tool (stdio)

## Status

Approved

## Story

**As an** instructor setting up the teaching server,
**I want** a Python MCP server project that starts over stdio and exposes a health/identity tool,
**so that** I have a runnable, connectable foundation and can confirm an agent is talking to the right server.

## Acceptance Criteria

1. A Python project is initialized with dependency management (e.g., pyproject/`uv` or `pip`), the official Python MCP SDK as a dependency, and a documented project structure that separates the MCP surface from the (future) mock data layer.
2. A unit + integration test harness is configured and runnable via a single command, with at least one passing smoke test.
3. The server runs over **stdio** via a single documented command and is discoverable by a standard MCP client.
4. The server exposes a health/identity tool (e.g., `get_server_info` / `ping`) that returns server name, version, and a static status, with a clear MCP schema and description.
5. An integration test connects an MCP client over stdio, invokes the health tool, and asserts the returned identity/status.
6. A brief README section documents how to install dependencies and run the server over stdio.

## Tasks / Subtasks

- [ ] Task 1: Initialize Python project scaffold (AC: 1)
  - [ ] Create `pyproject.toml` with project metadata, `requires-python = ">=3.11"`, entry point `contoso-support-mcp`, and dependencies (`mcp`, `pydantic`, `pyyaml`)
  - [ ] Create `src/contoso_support_mcp/` package with the layout from source-tree.md (tools/, prompts/, data/, fixtures/ placeholders)
  - [ ] Configure Ruff (lint+format) and pytest in `pyproject.toml`
  - [ ] Generate `uv.lock` via `uv sync`
- [ ] Task 2: Implement config + CLI entry point (AC: 3)
  - [ ] `config.py` with a `Settings` model (transport defaulting to stdio; fixtures path)

- [ ] Task 3: Implement the MCP server surface with the health tool (AC: 3, 4)
  - [ ] `server.py` builds the FastMCP app and registers tools
  - [ ] `tools/health.py` implements `get_server_info` returning `{name, version, status}` via a Pydantic model with clear schema/description
  - [ ] Wire stdio transport runner

  - [ ] Configure stdlib `logging` to **stderr only** — never stdout (stdout is the stdio transport channel)

## Dev Notes

### Technical Constraints
- **Language/runtime:** Python 3.12 was specified in architecture; **environment has 3.11.8**, so target `requires-python = ">=3.11"`. Non-blocking deviation, recorded. [Source: architecture/tech-stack.md]
- **MCP Framework:** official `mcp` Python SDK (FastMCP) — provides stdio and (later) streamable HTTP from one codebase. Pin exact version in `uv.lock` at init. [Source: architecture/tech-stack.md]
- **Data Validation:** Pydantic 2.x for models + tool I/O schemas. [Source: architecture/tech-stack.md]
- **Package/run manager:** `uv`. [Source: architecture/tech-stack.md]

### Critical Rules (from coding-standards.md)
- **No stdout writes except the MCP transport:** never `print()`; use the logger (stderr). A stray stdout write corrupts stdio-mode framing. [Source: architecture/coding-standards.md#Critical-Rules]
- **No randomness or wall-clock at query time.** [Source: architecture/coding-standards.md#Critical-Rules]
- **Tools return structured results, never raise past the handler.** [Source: architecture/coding-standards.md#Critical-Rules]

### File Locations (from source-tree.md)
- Entry/CLI: `src/contoso_support_mcp/__main__.py`
- Config: `src/contoso_support_mcp/config.py`
- Server: `src/contoso_support_mcp/server.py`
- Health tool: `src/contoso_support_mcp/tools/health.py`
- Tests: `tests/{unit,integration}/`
[Source: architecture/source-tree.md]

### Component Responsibilities
- **Transport Layer:** `run_stdio()` for offline student mode. [Source: architecture/components.md#Transport-Layer]
- **MCP Surface:** declares/handles tools; `get_server_info` health tool. [Source: architecture/components.md#MCP-Surface]
[Source: architecture/high-level-architecture.md]

### Testing
- **Frameworks:** pytest 8.x, pytest-asyncio for async client/server flows. [Source: architecture/test-strategy-and-standards.md#Unit-Tests]
- **Location/convention:** `tests/{unit,integration}/`, files `test_*.py`. [Source: architecture/coding-standards.md#Core-Standards]
- **This story:** unit smoke test for the health tool; integration test that connects an MCP client and invokes the health tool over the in-memory/stdio transport, asserting identity/status. [Source: architecture/test-strategy-and-standards.md#Integration-Tests]
- **Run command:** `uv run pytest`.
