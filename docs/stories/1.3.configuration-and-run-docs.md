# Story 1.3: Configuration & Classroom Run Documentation

## Status

Review

## Story

**As an** instructor or student preparing for a lab,
**I want** clear configuration and a single documented command per mode,
**so that** I can start the correct server for my scenario in under 10 minutes.

## Acceptance Criteria

1. Configuration (transport mode, host, port, and any core settings) is settable via a documented mechanism (CLI flags and/or a config file) with sensible defaults.
2. The documentation provides one copy-paste command to start **student-local stdio** mode and one to start **instructor-hosted network** mode.
3. The docs describe how a student confirms a successful connection using the health/identity tool.
4. Invalid configuration (e.g., a bad port) produces a clear, actionable error message rather than a stack trace.
5. The run docs are validated by following them from a clean environment to a successful health-tool call in both modes.

## Tasks / Subtasks

- [x] Task 1: Validate configuration inputs with clear errors (AC: 1, 4)
  - [x] Add port range validation (1–65535) and transport validation surfaced as a clean CLI error (exit code 2, no stack trace)
  - [x] Keep sensible defaults (stdio; 127.0.0.1:8000)
- [x] Task 2: Consolidate run documentation (AC: 2, 3)
  - [x] README "Quickstart" with one copy-paste command per mode and the health-tool confirmation step
- [x] Task 3: Tests for config validation (AC: 4)
  - [x] Unit tests: invalid port (0, 70000, non-int) and invalid transport produce a `SystemExit`/clean error, not a traceback

  - [x] Script/documented steps: from `uv sync` to a successful `get_server_info` call in both modes (evidence captured in Dev notes)

- **Health confirmation:** `get_server_info` (from 1.1). [Source: architecture/components.md#MCP-Surface]

[Source: architecture/source-tree.md]

### Previous Story Insights (1.1, 1.2)
- CLI already parses `--transport {stdio,http}`, `--host`, `--port`. This story hardens validation + docs; argparse `choices` already rejects bad transport with exit 2.
- Port is currently `type=int` with no range check — add explicit range validation.

### Testing
- **Frameworks:** pytest. [Source: architecture/test-strategy-and-standards.md]
- **This story:** unit tests asserting invalid config exits cleanly (SystemExit / non-zero) without a traceback; verify defaults. [Source: architecture/test-strategy-and-standards.md#Unit-Tests]
- **Run:** `uv run pytest`.
