# Story 5.1: Known-Issues KB Tool

## Status

Draft

## Story

**As a** student's AI agent,
**I want** to search a read-only Azure known-issues knowledge base,
**so that** I can look up generic remediation guidance for a symptom without it revealing the ticket-specific answer.

## Acceptance Criteria

1. A `search_known_issues` tool accepts optional `query` (keyword), `product`, and `category` filters (AND semantics) and returns matching known-issue entries (title, product, category, symptom, remediation, optional doc link).
2. The KB is a **separate curated dataset** (≥ 12 generic entries across ARM/Network/Compute-host/Compute-guest and the three products) that does NOT copy any per-scenario root cause/resolution — it cannot leak the grading key.
3. An invalid `category` returns a structured `invalid_request`; no matches returns `ok` with an empty list; results are deterministic.
4. The tool exposes a clear MCP schema/description stating it is generic guidance, not ticket-specific RCA.
5. Unit + integration tests cover loading/validation, each filter, keyword search, no-match, and invalid category; the KB loads at startup via the same fail-fast pattern as scenarios.

## Tasks / Subtasks

- [ ] Task 1: KB data + model (AC: 2)
  - [ ] `fixtures/known_issues.yaml` — top-level `known_issues:` list, ≥12 generic entries (arm×4, network×3, compute_host×3, compute_guest×3) across the 3 products; each `{id: KB-<DOMAIN>-###, title, product, category, symptom, remediation, doc_link}`. No per-ticket resolution text.
  - [ ] `data/models.py` — add `KnownIssue` (strict; `category: RootCauseCategory`; `doc_link: str | None`); `KNOWN_ISSUE_ID_RE`.
- [ ] Task 2: Loader + config (AC: 5)
  - [ ] `config.py` — `DEFAULT_KNOWN_ISSUES_PATH`.
  - [ ] `data/loader.py` — `load_known_issues(path) -> list[KnownIssue]` (fail-fast, unique-id, id-sorted).
- [ ] Task 3: Repository + tool (AC: 1, 3, 4)
  - [ ] `data/repository.py` — `Repository(dataset, known_issues=None)`; `search_known_issues(*, query, product, category)` (AND; case-insensitive substring over title+symptom+remediation).
  - [ ] `tools/kb.py` — `register_kb_tools(mcp, repo)`; `search_known_issues(query=None, product=None, category=None)`; validate category → `invalid_request`; empty → `ok`.
  - [ ] `server.py` — load KB + register; startup log includes KB count.
- [ ] Task 4: Tests (AC: 5)
  - [ ] `tests/unit/test_known_issues.py` (loader + repository + model negatives).
  - [ ] `tests/integration/test_kb_tools.py` (no-arg, category filter, product filter, keyword hit, no-match, invalid category).

## Dev Notes

### Design [Source: architecture/data-models.md#KnownIssue-KB-Epic-5] [Source: architecture/workshop-integration-epic-5.md]
- KB is a curated dataset **decoupled** from `scenario.root_cause` — must NOT copy `resolution` text (grading key is `docs/scenario-index.md`). [Source: prd/epic-5-workshop-integration.md] (Story 5.1 AC2)
- Mirror existing patterns: loader like `data/loader.py::load_scenarios`; tool like `tools/tickets.py`; result model with `status ∈ {ok, invalid_request}` (no `not_found` — a search that matches nothing is `ok` + empty list). [Source: architecture/error-handling-strategy.md#Business-Logic-Errors]
- `category` validated against `RootCauseCategory` (arm/network/compute_host/compute_guest); `product` is free-text (unknown product → empty ok). [Source: architecture/data-models.md]
- Repository constructor gains an optional `known_issues=None` arg so all existing call-sites/tests keep working. Deterministic (id-sorted source list). [Source: architecture/components.md#Repository]

### File Locations [Source: architecture/source-tree.md] [Source: architecture/workshop-integration-epic-5.md]
- `src/contoso_support_mcp/fixtures/known_issues.yaml` (sibling of `scenarios/`; scenario glob `scenarios/*.yaml` ignores it)
- `src/contoso_support_mcp/data/{models.py,loader.py,repository.py}`, `config.py`, `tools/kb.py`, `server.py`
- Tests: `tests/unit/test_known_issues.py`, `tests/integration/test_kb_tools.py`

### Testing
- pytest + in-memory MCP client (as in `tests/integration/test_ticket_tools.py`). Assert `search_known_issues({"query":"allocation"})` hits `KB-ARM-001`; `{"category":"martian"}` → invalid_request; `{"query":"zzz"}` → ok empty.
- Grep tests for any hardcoded tool count (surface 9→10).
- **Run:** `uv run pytest`.
