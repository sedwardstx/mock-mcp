# Story 3.5: Telemetry Backfill & Multi-Round Scenario Authoring

## Status

Approved

## Story

**As an** instructor,
**I want** existing scenarios backfilled with correlated telemetry and new multi-round scenarios added,
**so that** agents can perform full RCA and some incidents require iterative investigation.

## Acceptance Criteria

1. All seed scenarios from Epic 2 are backfilled with correlated telemetry across the relevant tables, so each has an evidence trail supporting its defined root cause.
2. Additional scenarios are authored to reach a cumulative total of ~60 distinct scenarios, spanning ARM/Network/Compute root causes across both personas.
3. At least a handful of scenarios are **multi-round**: the ticket symptom alone is insufficient, and the supporting evidence is distributed such that an agent must query telemetry, form a hypothesis, and query again (a different table, time range, or resource/instance) to reach the root cause.
4. For each multi-round scenario, the intended investigation path (which queries reveal which evidence) is documented to support lab authoring and grading.
5. The consistency check is extended to verify, for every scenario, that the telemetry actually contains evidence consistent with the defined root cause; it runs in the test suite and fails on any inconsistency.
6. End-to-end tests demonstrate at least one single-round and one multi-round scenario solved via the tool sequence (ticket → resource → telemetry → root cause).

## Tasks / Subtasks

- [ ] Task 1: Telemetry-aware generator (AC: 1, 2)
  - [ ] Extend `scripts/author_seed_scenarios.py` to emit category-correct telemetry (evidence row + benign noise) for every generated scenario; grow the curated/generated set to ~60 total
- [ ] Task 2: Multi-round scenarios (AC: 3, 4)
  - [ ] A handful of `multi_round` scenarios where the "obvious" table is inconclusive and the real evidence lives in another table/instance; 2-step `investigation_path` documented
- [ ] Task 3: Consistency: telemetry evidence (AC: 5)
  - [ ] Extend `check_dataset` so each scenario's root_cause.category has matching evidence in the correct table for its resource; negative test
- [ ] Task 4: End-to-end tests (AC: 6)
  - [ ] `tests/e2e/`: single-round RCA (ticket→resource→telemetry→cause) and a multi-round RCA (first query inconclusive, second reveals cause)

## Dev Notes

### Evidence Heuristics (per category) [Source: architecture/database-schema.md]
- `arm`: an ArmTraceRow with `activity_status == "Failed"` (or http_status_code ≥ 400).
- `network`: a NetworkLogRow with `action == "Deny"`.
- `compute_host`: a ComputeHostLogRow with `health_status in {"Degraded","Unavailable"}` or `level == "Error"`.
- `compute_guest`: a ComputeGuestLogRow with `level in {"Error","Warning"}` (Windows-only).

### Multi-Round Design [Source: prd/epic-3-kusto-telemetry-tools-correlation.md]
- The "obvious" first query (a table implied by the symptom) returns only benign rows; the true evidence is in a different table/instance. `investigation_path` has 2 steps: step 1 (obvious, inconclusive), step 2 (reveals the cause). `difficulty: multi_round`.

### Generator [Source: Story 2.5]
- The generator now owns telemetry for scenarios 4..60 (single source of truth). Hand-authored 1–3 keep their telemetry. Re-running the generator is deterministic (no clock/random).

### Consistency Extension [Source: architecture/test-strategy-and-standards.md#Consistency-Tests]
- `check_dataset` gains a per-scenario evidence check (category → table → evidence row for the resource). Keeps prior ticket↔resource checks.

### File Locations [Source: architecture/source-tree.md]
- `scripts/author_seed_scenarios.py` (rewrite — telemetry + multi-round + reach ~60)
- `src/contoso_support_mcp/fixtures/scenarios/*.yaml` (regenerated 4..60)
- `src/contoso_support_mcp/data/consistency.py` (extend)
- Tests: `tests/e2e/` (new), extend consistency tests

### Testing
- pytest. Consistency suite must be green over all ~60. E2E drives the actual MCP tools for one single-round and one multi-round scenario.
- **Run:** `uv run pytest`.
