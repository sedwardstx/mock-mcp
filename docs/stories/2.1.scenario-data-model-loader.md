# Story 2.1: Scenario & Ticket Data Model + Deterministic Loader

## Status

Review

## Story

**As a** server developer,
**I want** a well-defined scenario/ticket/resource data model with a deterministic fixture loader,
**so that** all tools read from one consistent, extensible source and later scenarios can be added without code changes.

## Acceptance Criteria

1. A documented data model defines a **scenario** and its linked entities: ticket (id, product/service, symptom/problem statement, severity, status, customer persona, timestamps), affected Azure resource(s) and subscription, references to correlated telemetry (populated in Epic 3), and a defined root cause + resolution/next-step.
2. Ticket identifiers conform to `TICKET-XXXXXXXX` (8 digits) and are unique across the dataset.
3. A loader reads scenarios from embedded fixture files (e.g., JSON/YAML) into in-memory structures at startup, with no external database.
4. Loading is **deterministic** â€” the same fixtures yield the same in-memory data and the same tool responses every run (no query-time randomization).
5. The loader validates fixtures on load and fails fast with a clear error identifying the offending scenario/field if the schema is violated.
6. A small set of representative sample scenarios (enough to develop and test the ticket tools) is included and loads successfully.
7. Unit tests cover the loader, schema validation (including a negative case), and the determinism guarantee.

## Dev Notes

### Data Model (authoritative shape) [Source: architecture/data-models.md] [Source: architecture/database-schema.md#Scenario-Fixture-File]
- **Scenario:** `scenario_id` (mirrors ticket id), `difficulty` (`single_round`|`multi_round`), `ticket`, `resources[]`, `telemetry` (four tables; empty until Epic 3), `root_cause`, `investigation_path[]`.
- **Ticket:** `ticket_id` (`TICKET-XXXXXXXX`), `title`, `symptom`, `azure_product`, `persona` (`windows_admin`|`azure_developer`), `severity` (`Sev1..4`), `status` (`New`|`Active`|`Pending`|`Resolved`), `created_at`/`updated_at` (ISO 8601), `resource_ids[]`.
- **AzureResource:** `resource_id` (ARM id), `resource_type` (e.g. `Microsoft.Compute/virtualMachines`, `.../virtualMachineScaleSets`), `name`, `resource_group`, `subscription_id`, `location`, `instances[]` (VMSS; empty for VM).
- **RootCause:** `category` (`arm`|`network`|`compute_host`|`compute_guest`), `summary`, `resolution`.
- **InvestigationStep:** `order`, `tool`, `params`, `reveals`.

### Constraints
- **No database**; embedded YAML fixtures loaded into memory at startup. [Source: architecture/high-level-architecture.md]
- **Determinism:** no `random`/`datetime.now()` affecting loaded data or ordering. [Source: architecture/coding-standards.md#Critical-Rules]
- **Fail-fast validation** at load; runtime data immutable. [Source: architecture/error-handling-strategy.md#Data-Consistency]
- **Scenarios are data, not code** â€” new scenarios require no server changes. [Source: architecture/coding-standards.md#Critical-Rules]
