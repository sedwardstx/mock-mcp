# Story 1.2: Network Transport Mode (Instructor Hosting)

## Status

Draft

## Story

**As an** instructor hosting the server for the class,
**I want** to run the same server over a network transport,
**so that** many students can connect to a single instructor-hosted endpoint.

## Acceptance Criteria

1. The server supports a **network transport** (streamable HTTP/SSE) exposing the identical tool surface as stdio mode, from the same codebase.
2. The transport mode is selectable at startup (CLI flag and/or config), with host and port configurable.
3. A remote MCP client can connect over the network transport, invoke the health/identity tool, and receive the correct response.
4. The server accepts multiple concurrent client connections and responds correctly to each (validated by a test simulating concurrent clients).
5. An integration test exercises the health tool over the network transport.
6. No external services, credentials, or internet access are required to run the network mode.

## Tasks / Subtasks

- [ ] Task 1: Add network transport selection to CLI/config (AC: 1, 2)
  - [ ] Extend `--transport` choices to include `http` (streamable HTTP); add `--host` and `--port` args with defaults (127.0.0.1:8000)
  - [ ] Thread host/port into the FastMCP server settings
- [ ] Task 2: Run streamable HTTP from the same server build (AC: 1, 6)
  - [ ] `__main__.py` dispatches `http` → `mcp.run(transport="streamable-http")`; identical tool surface (no separate registration path)
  - [ ] Confirm no external/network dependencies beyond the local bind
- [ ] Task 3: Integration test over the network transport (AC: 3, 5)
  - [ ] Start the server (uvicorn) on an ephemeral localhost port in a background thread
  - [ ] Connect via the MCP streamable-HTTP client; call `get_server_info`; assert identity
- [ ] Task 4: Concurrency test (AC: 4)
  - [ ] Open multiple concurrent client sessions against the running server; assert each gets a correct response
- [ ] Task 5: README network-mode docs (AC: 2)
  - [ ] Document the instructor-hosted command with host/port

## Dev Notes

### Technical Constraints
- **Transport:** streamable HTTP/SSE via FastMCP `mcp.run(transport="streamable-http")`, same codebase as stdio. [Source: architecture/tech-stack.md] [Source: architecture/components.md#Transport-Layer]
- **ASGI server:** Uvicorn hosts the HTTP app. [Source: architecture/tech-stack.md]
- **Config:** transport/host/port selectable at startup. [Source: architecture/components.md#Config-CLI]
- **Concurrency:** read-only, stateless responses → naturally concurrency-safe for classroom scale. [Source: architecture/high-level-architecture.md] [Source: prd/requirements.md] (NFR4)
- **Offline:** binds a local port only; no external services/credentials. [Source: prd/requirements.md] (NFR1)
- **Auth:** none in MVP; open endpoint on trusted LAN (confirmed). [Source: architecture/security.md#Authentication-Authorization]

### File Locations
- CLI/config: `src/contoso_support_mcp/__main__.py`, `config.py`
- Server (unchanged surface): `src/contoso_support_mcp/server.py`
- Tests: `tests/integration/test_server_http.py`
[Source: architecture/source-tree.md]

### Previous Story Insights (1.1)
- FastMCP built via `build_server()`; health tool `get_server_info` returns structured `ServerInfo`. Reuse the same build for HTTP — do not fork the surface.
- Keep all logging on stderr (Critical Rule). [Source: architecture/coding-standards.md#Critical-Rules]
- mcp SDK is 1.28.1; verify streamable-HTTP client API by introspection before writing the test (carried-forward gate note).

### Testing
- **Frameworks:** pytest, pytest-asyncio. [Source: architecture/test-strategy-and-standards.md]
- **This story:** integration test over a real localhost HTTP bind (background uvicorn) + a concurrency test with multiple simultaneous sessions. [Source: architecture/test-strategy-and-standards.md#Integration-Tests]
- **Run:** `uv run pytest`.
