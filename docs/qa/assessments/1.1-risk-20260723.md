## Detailed Risks & Mitigations

### TECH-001 — MCP SDK transport/API mismatch (Score 4, Medium)
- **Probability:** Medium — the SDK evolves; exact FastMCP `run()` / client API must be verified.
- **Impact:** Medium — a wrong API blocks the health tool and the integration test, i.e., the whole story.
- **Mitigation (preventive):** Verify the installed SDK's API by introspection before writing the integration test; pin the exact resolved version in `uv.lock`.
- **Testing focus:** Integration test that actually connects a client and calls the tool proves the API is correct.

### OPS-001 — stdout contamination (Score 3, Low)
- **Probability:** Low — but a single stray `print()` anywhere in `src/` silently breaks stdio.
- **Impact:** High — the server appears to start but the protocol is corrupted; hard to diagnose in a classroom.
- **Mitigation (preventive + detective):** Route all logging to stderr; add a guard test that fails if `print(` appears in `src/`.
- **Testing focus:** `1.1-UNIT-003` (no-stdout guard).

### TECH-002 — Python version deviation (Score 1, Minimal)
- **Mitigation:** `requires-python = ">=3.11"`; recorded in Dev Notes. No action beyond documentation.

### OPS-002 — Reproducibility (Score 1, Minimal)
- **Mitigation:** Commit `uv.lock` for reproducible classroom installs.
